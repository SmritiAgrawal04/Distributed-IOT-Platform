A Distributed IOT Platform

A server-less IoT solution development and management platform offers core functionality for managing the entire lifecycle of IoT solution development. This project is an attempt to to build a Distributed IoT platform that enables users to write programs and algorithms that use it in various domains of automation.

The platform consists of a Django based web framework UI; and various modules like- Scheduler, Load Balancer, Monitoring service, Logging service, Deployer, Action and Notification Manager, Databases, Runtime Servers.

A BRIEF DESCRIPTION OF THE PLATFORM SERVICES:

Account Manager

The accounts module of my platform allows a user to register/login. When an already registered user logs in, she is directed to either her app developer profile or app user profile as per her role. Logging out of the profile is also managed by Accounts only. On the app developer profile lies all the sensor information sorted based on the both location and its type, the developer can conveniently choose her requirements to build her application. Finally, she can upload her developed app consisting of the services. On the app user profile lies a list of all the applications uploaded to my platform and the user can see all descriptions given by the uploader about the app and its services and schedule required services.

Request Manager

All the requests from the app user made on the User Interface are managed by Request Manager. It processes each request and serves it.

Scheduler

This services makes sure that timely scheduling of the user requests happen.

Monitoring Module

The Monitoring Module is responsible to collect the current system stats from all Runtime Servers and pass them all to the Load Balancer.

Load Balancer

To have a balance in the system such that it isn’t only one server that runs all the user services at all times but the load is distributed, load balancing is indispensable. Under this service, I’ve looked into the requirements of the service as mentioned by the app dev and compared them to the stats of all the servers. Finally, fetched the server which is closest to the need of the service and returned the server details to the scheduler.

Deployer

On having received the details on which the service will run, it's the responsibility of the deployer to make it ready with all the dependencies installed to avoid any error when the service executes.

Logging Module

The Logging Module maintains a log of every user request that is being scheduled to be fault tolerant to any failures. It keeps the information like- application name, service name, user details, scheduling type and time, runtime server details etc. in a structured database.

Sensor Manager

It manages all sensor information and communicates with them to collect sensor readings.

Sensors

These are dummy files representing sensors and producing the data on sensor manager topics.

Runtime Servers

These are the systems which actually execute the service after receiving all the service details.

Action & Notification Service

Its responsibility is to notify the user for the actions happening in her scheduled service via messages,emails or alarm as the urgency of the service is designed.

Health Checkup Module

It keeps a check on the services/modules and the runtime services if they are up and running.
